# タスク: Runtrip人気リサーチ由来の新規コース100件整備(バッチ1)

Runtripリサーチ(`data/research/runtrip-popular.json`、1,851件)から選定した100件を、**自前GPX**でドコランの新規スポットとして整備する。末尾の進捗表の上から順に、1件ずつ処理すること。

## 最重要: Runtripデータの使い方の線引き

- Runtripのデータは**「その行き先が人気だという事実」と「ルート特定のヒント」としてのみ**使う
- **禁止**: Runtripのタイトル・説明文の転用(言い換えでの流用も不可)、画像の取得・転用。スポット名は事実ベースで自分で付ける(例: Runtrip「雲の上を走る!? 美の山公園散策ランコース」→ ドコラン「美の山公園」)
- 目的は**Runtripコースの複製ではない**。「その行き先の代表コースを自前で設計する」こと。Runtripの距離は目安であり、定番ルートとして自然な距離に調整してよい(調整したら進捗表の状態欄に実距離を記録)
- コース形状はBRouter(OSM)で自前生成する。Runtripのルートデータはそもそも取得していないし、使わない

## 選定の背景(2026-07-10)

- 選定元: 非重複(既存スポットから2km超)1,530件 → スタート地点800m以内クラスタの代表1,017件
- **バケットA(34件)**: 全国人気(行きたい数150以上)。関東のトレイル・ロングが中心
- **バケットB(66件)**: 掲載3件未満の県の底上げ。完了すると**全47都道府県が掲載3件以上**になる(現在: 掲載0の県が8つある)
- 各候補の説明文などの詳細は `data/research/runtrip-popular.json` を候補ID(進捗表の「候補ID」列)で引く

## 前提・環境

- パッケージ管理は **npm**(pnpmはこのマシンで壊れている)。DBは**ローカルD1**(miniflare。`getDb()`が自動で解決する。Docker不要)
- 他のエージェントが同リポジトリを並行編集することがある。編集前に対象ファイルの最新状態を確認する
- BRouterは無料の公共サービス。**リクエスト間に2〜3秒スリープ**、連続失敗したら中断
- 本タスクでは**本番D1への書き込みはしない**(最後の「本番投入」節を参照。投入はユーザー確認後に別途実行)

## 1件の処理手順

### 1. ルートを特定する

`data/research/runtrip-popular.json` から候補IDで引いて `title` / `description` / `startLatitude,startLongitude` / `distance` を読む(ヒントとしてのみ。文章の転用禁止)。地図・Web検索で実在のコース(周回路・遊歩道・トレイル・河川敷など)を特定する。

- **特定できない・実在の裏付けが弱いものは、でっち上げずに進捗表へ「skip(理由)」と書いて次へ**。skipは失敗ではない(過去バッチでも21/25などが通常)
- スタート座標はRuntrip申告値。駅・公園入口など実際の起点とずれていることがあるので、Nominatim/地理院地図で照合して自分の起点を決める

### 2. スポット名・slugを決める

- スポット名: 場所の正式名称ベース(「◯◯公園」「◯◯湖」「◯◯緑道」等)。トレイルは「◯◯トレイル(起点名)」のような事実ベースの名前
- slug: ローマ字ケバブケース(例: `minoyama-park`)。既存slugと重複しないこと
- nameKana(ひらがな)・city(市区町村)も必要(db:importスキーマ準拠)

### 3. BRouterでGPXを生成する

`docs/gpx-course-task.md` の手順2〜3と同じ。要点:

```bash
curl -s --max-time 30 "https://brouter.de/brouter?lonlats=<lng1>,<lat1>|...|<lng1>,<lat1>&profile=shortest&alternativeidx=0&format=gpx" -o data/gpx/<slug>.gpx
```

- 経由点4〜7点を実在の道路・園路・登山道上に置く。周回は最初と最後を同座標に、往復は 起点→折返し→起点
- ロードは `profile=shortest`、トレイルは `profile=hiking-mountain` も試す。`target island detected` は経由点を通行可能な道にずらす

### 4. 検証する

```bash
npm run gpx:check -- data/gpx/<slug>.gpx
```

- `distanceKm` が期待距離(進捗表の距離km)の**±15%以内**(トレイルのみ±20%まで可)。意図的に距離を変えた場合はその値を基準にし、表に記録
- `courseType` が実態と一致、`elevationGainM` が常識的
- 座標列を目視確認(スタート付近の棘・コース外への膨らみがないか)

### 5. スポットとして登録する(ローカル)

10件程度ごとにバッチJSON(`data/spots-runtrip-b1-<連番>.json`)を作り、db:importスキーマ(`src/db/import.ts` 参照)で記述する:

- `description`: **50字以上、完全に自分の言葉**。自分で調べた事実(経由地・路面・高低差・信号・トイレ・補給・アクセス)を書く。Runtrip説明文の言い換えは不可
- `tags`: 既存タグのみ(`SELECT slug FROM tags` で確認)
- `course.distanceM` は gpx:check の実測値

```bash
npm run db:import -- data/spots-runtrip-b1-<連番>.json   # is_published=true で入るので、QC合格分だけJSONに入れること
npm run gpx:apply -- <slug> data/gpx/<slug>.gpx
```

### 6. サムネ・OGPを生成して目視QC

```bash
npx tsx scripts/generate-course-maps.ts <slug...>
npx tsx scripts/generate-og-images.ts <slug...>
```

サムネを見て、コース形状が不自然(直線ワープ・棘・孤立線)なら手順3からやり直し。直らなければ `UPDATE spots SET is_published = false WHERE slug = '<slug>';` をローカルで実行し、進捗表に「review(理由)」と記録

### 7. 進捗表を更新する

状態欄: `done` / `done(実測◯km)` / `skip(理由)` / `review(理由)`。slug欄に決定slugを記入

## 本番投入(全件処理後・ユーザー確認後)

ローカルD1で整備した新規行(spots / courses / spot_tags)をINSERT文にエクスポートし、`node scripts/run-sql.mjs <file.sql> --remote` で本番D1へ投入する。**D1はSQL 1文100KBの上限がある**ため:

- 1行=1 INSERT文に分割。geojsonが大きい長距離コースは、100KBを超えないようトラックポイントを間引いてから gpx:apply し直す(分割手法は `scripts/export-neon-to-d1.ts` が参考になる)
- 投入前にローカルで同じSQLを流して検証する
- 投入後 `npm run deploy` は不要(次回デプロイのpredeployで本番スナップショットに反映される)が、サムネ/OGP画像(public配下)はデプロイで配信されるため、投入とデプロイはセットで行う

## 合格基準(バッチ全体)

- 進捗表100件すべてに状態が記入されている(done / skip / review)
- done率の目標は7割(70件)以上。skipが3割を超えたら中断してユーザーに報告
- done分はすべて: gpx:check合格、サムネ目視QC済み、description 50字以上の自分の言葉、既存スポットと500m以内の重複なし(db:importの警告に対応済み)
- バケットB完了後、全47都道府県で掲載3件以上(`SELECT prefecture, COUNT(*) FROM spots WHERE is_published = 1 GROUP BY prefecture` で確認)
