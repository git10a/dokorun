# タスク: 駅からラン(駅ページ + ひと駅ラン)

「大岡山駅で降りたらどこ走る?」に答える駅単位のページ群を作る。駅ごとに (1) 駅チカのスポット、(2) 同じ路線で数駅先まで足を伸ばせば走れるスポット(=ひと駅ラン)を見せ、隣駅ページへ辿れるようにする。SEO面としては「◯◯駅 ランニング」「◯◊駅 走る」の検索面を新規に取りにいく。

## フェーズ分け

- **Phase 1(このタスクの本体)**: 駅マスタ整備 → スポットとの紐付けパイプライン → 駅ページSSG → サイトマップ・内部リンク
- **Phase 2**: スポットページに「最寄り駅」チップ(駅ページへの逆リンク)、トップ/エリアページからの導線
- **Phase 3(別タスク)**: ログインユーザーの最寄り駅登録と、トップでのパーソナライズ表示

## 背景・既存資産

- スポットはD1 `spots` テーブル(公開約228件、`lat`/`lng`/`slug`/`name`/`prefecture`/`city` あり)。ビルド時は predeploy が本番D1スナップショットを生成する仕組みがある
- 駅名→座標の解決APIが既にある: `src/app/api/stations/resolve/route.ts`(HeartRails)。ただしこれは単発クエリ用で、ページの静的生成・隣駅グラフには使えない → 駅マスタを別途持つ
- スポットカードは `src/components/spot-card.tsx` を再利用する
- サイトマップは `src/app/sitemap.ts`

## 制約(重要)

- **Workerバンドルは上限3072KiBに対し既に約2400KiB**。駅マスタJSONを `import` でバンドルに入れてはいけない。駅ページは完全静的(`generateStaticParams` + fallbackなし、`dynamicParams = false`)にして、駅データはビルド時に `fs` で読む。プリレンダー済みHTMLはassets配信になるのでバンドルを膨らませない。ビルド後にワーカーサイズを必ず確認すること
- パッケージ管理はnpm(pnpm不可)。deployはnode@23で `npm run deploy`。新規checkoutでは `npm run cf-typegen` が必要
- 他エージェントが並行編集していることがある。既存ファイルの大改造は避け、追加中心で

## 駅マスタのデータ調達

**駅データ.jp(ekidata.jp)のCSVを使う**(要無料会員登録。利用規約を実装前に確認し、必要な出典表記を about ページ等に入れる):

- `station.csv`: 駅コード `station_cd`、駅グループ `station_g_cd`(同名駅の会社跨ぎ統合)、駅名、かな、路線コード `line_cd`、都道府県、経度緯度
- `line.csv`: 路線名
- `join.csv`: **隣接駅ペア**。これがあるので「1駅先・2駅先」のグラフ探索がそのまま組める(国土数値情報N02には隣接順がないため不採用)

CSVは `data/ekidata/` に置き、gitにコミットする(更新は年1回程度の手動でよい)。

## パイプライン: scripts/build-station-data.mjs

入力: `data/ekidata/*.csv` + 公開スポット一覧(ビルド時スナップショットと同じ経路で slug/name/lat/lng/prefecture を取得)。

処理:

1. `station_g_cd` で駅を統合(1駅=1ページ。所属路線は配列で持つ)
2. 各スポットについて全駅との直線距離(ハバースイン)を計算し、**2.0km以内**の駅を「駅チカ」として紐付け。徒歩分数は `距離m / 80` を切り上げ
3. `join.csv` の隣接グラフをBFSし、各駅から**3駅以内**で到達できる駅の駅チカスポットを「ひと駅ラン」候補として集める(経由路線名と駅数を保持)
4. **ページ生成対象の駅** = 駅チカ1件以上、または ひと駅ラン候補2件以上。それ以外の駅は出力に含めない(薄いページを量産しない)
5. slugを生成: かな→ヘボン式ローマ字(例: おおおかやま → ookayama)。重複時は都道府県ローマ字をサフィックス(`ookayama-tokyo`)。それでも衝突したら `station_g_cd` を付ける。**slugは一度公開したら変えない**ので、生成結果を `data/stations/slugs.json` に固定保存し、再実行時は既存slugを優先する

出力:

- `data/stations/stations.json` — ページ生成対象駅のみ。駅ごとに: slug、名前、かな、都道府県、lat/lng、路線名リスト、駅チカスポット(slug・距離m・徒歩分)、ひと駅ランスポット(slug・経由路線・駅数・降車駅名・降車駅からの徒歩分)、隣接するページ対象駅のslugリスト(路線名付き)
- 想定規模: 数百〜2000駅程度。1駅あたり数百バイトに収める(スポット側の名前等は駅データに複製せず、slugで参照してページ側でスポット情報を引く)

## ページ: /stations/[slug]

`src/app/stations/[slug]/page.tsx`。完全静的生成。データ読み込みは `src/lib/stations.ts`(ビルド時 `fs` 読み、`import` 禁止)。

構成:

1. **H1**: 「◯◯駅から走れるランニングスポット」。メタ: title 同等 + サイト名、description は駅チカ件数と代表スポット名を含む文
2. **駅情報**: 路線名チップ、都道府県
3. **地図**: 駅マーカー + 駅チカスポットのピン(既存の地図コンポーネント流用。重ければPhase 1では省略可)
4. **駅チカセクション**: 徒歩分の近い順に SpotCard + 「徒歩◯分」バッジ
5. **ひと駅ランセクション**: 「◯◯線で1駅・△△駅下車 徒歩◯分」のように、乗車駅数の少ない順にグループ表示。ここが「1駅行ってから走る」体験の核
6. **隣の駅ナビ**: 路線ごとに前後のページ対象駅へのリンク(「◯◯線でとなりの駅: △△駅」)。ページ対象でない隣駅はリンクしない
7. **パンくず**: ホーム > 都道府県エリアページ > 駅

一覧ページ `/stations/page.tsx` は都道府県別の駅リンク集(1ページで可、駅数が多ければ都道府県アコーディオン)。

## Phase 3 の方向性だけメモ(実装しない)

- `user` テーブルに `home_station_slug` を追加、/me で登録(入力補完は `public/stations/index.json` = ページ対象駅の name/kana/slug の軽量索引をクライアント取得)
- トップに「◯◯駅からのおすすめ」セクション。駅ページと同じデータを `public/stations/<slug>.json` として出力しておきクライアント取得すれば、Workerバンドルを増やさず実現できる

## 検証

1. `npm run build` が通り、駅ページがプリレンダーされること(ビルドログの Static 印を確認)
2. ワーカーサイズが上限内に収まっていること(ビルド出力のサイズ表示で確認。増分が数十KiBを超えるなら設計を疑う)
3. 大岡山駅・代々木駅など具体例で: 駅チカと「ひと駅ラン」の中身が地理的に正しいこと(洗足池公園が大岡山から近い、等を目視確認)
4. 同名駅(例: 府中、大久保)が都道府県サフィックスで衝突なく生成されること
5. `npx tsc --noEmit` とESLintが通ること
